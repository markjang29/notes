# ADR — 쿼터/검색 장애 대응 프로토콜

## 상태
proposed → (사용자 합의 후 accepted)

## 날짜
2026-06-26

## 프로젝트 / 적용 범위
공통 (모든 담당 봇 — 매니저·팀장·subagent)

## 결정
LM 호출이 **쿼터/과부하로 실패** (HTTP 529 "service overloaded", MCP -429 rate limit, search API "busy" 등)하면:

1. **즉시 재시도 금지** — 쿼터 구간엔 재실패만 반복·토큰만 소모.
2. **체크포인트 기록** (영구 — `~/notes`):
   - 현재 목표 · 실패한 단계 · 완료분 · 다음 스텝 · 산출물 파일 경로.
   - 필드는 `personas/markjang29.md` §1(완성 판정) + `principles/context-budget.md` §3(인계 필드) 준용.
3. **예약 재개** — `cokacdir --cron "<재개 프롬프트>" --at "<쿼터 리셋 예상시각>" --session <SID> --once`:
   - `--session` 으로 같은 세션 맥락을 이어 받아 중단 단계부터 계속.
   - 리셋 예상시각 = 마지막 성공 호출 시각 + 쿼터 윈도우(현재 5h).

## 맥락
- Z.AI / Codex 가 5시간 쿼터제 → 세션 중간에 LM 호출 차단 빈번.
- 아이디에이션 v1 작업 중 팀장들이 **병렬 WebSearch 과다** → search API busy(MCP -429)·서버 과부하(529) 반복. 일부 subagent 529로 중단 후 맥락 손실.
- 인계(checkpoint) 없이 세션이 끊기면 다음 세션이 *단절 지점*을 복원 못 함 → 의도부채(`ai-dev-신념.md` §3) 위험.

## 제약
- 봇이 자기 쿼터 잔량을 직접 알기 어려움 → 실패 응답으로 간접 감지.
- 5h 윈도우 리셋 시각은 추정치(마지막 성공 + 5h).
- `--session` 재개는 같은 봇·같은 세션 ID에서만 동작.

## 선택한 이유 / 버린 대안
- **선택:** 체크포인트(영구 notes) + `--cron --session` 예약 재개. 영구 기록 + 자동 재개를 동시에.
- 버린 — **그 자리에서 무한 재시도**: 쿼터 동안 계속 실패, 토큰만 소모, 맥락만 낡음.
- 버린 — **수동 재개만**: 사용자가 다시 불러야 함. Telegram 은 비대화형이라 지연 큼.
- 버린 — **인계 없이 세션 종료**: 단절 지점·부분 산출물이 사라짐.

## 트레이드오프
- **얻은 것:** 쿼터 장애에도 진행 맥락 보존 + 크론으로 자동 재개 + 부분 산출물 보존.
- **잃은 것:** 체크포인트 작성 오버헤드 + 크론 스케줄 관리 부담.

## 검증 기준
- 다음 쿼터 장애 시 (a) 체크포인트가 `~/notes`에 남고, (b) 리셋 후 크론이 같은 세션에서 해당 단계를 재개하는지.
- 재개 시 단절 전 맥락(목표·파일 경로·다음 스텝)이 손실 없는지.

## 실패 / 복구 과정
- **이번 세션 사례:** 팀장들이 WebSearch 를 한 번에 다수 병렬 발행 → search API busy(MCP -429)·529 반복. 일부 subagent 중단.
- **교훈 — 검색 병렬도 제한:**
  - 동시 검색 **권장 1~2회**. 그 이상은 순차 처리.
  - 429/529 수신 시 즉시 중단 → 체크포인트 → (위 결정3) 예약 재개.
  - 재시도는 지수 백오프(예: 30s → 2m → 10m), 3회 초과 시 예약 재개로 전환.

## 후속 재검토 조건
- 쿼터 정책 변경(5h → 다른 윈도우) 시 재개 스케줄 재설계.
- 검색 API rate limit 변동 시 병렬도 임계 조정.
- cokacdir `--session` 크론 동작 검증 후 accepted 로 승격.
