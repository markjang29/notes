## 확정 설계

**① 디벨롭 구조 (연재형)**
```
experiment  ← 시드 (주제 + 초기 캐릭터 + 플롯 개요)
   ↓
chapter     ← N장 (1장, 2장, 3장...) — experiment에서 뻗어나감
   ↓
scene       ← 각 장의 전투/대화/탐색 씬들
```

experiment 1개가 **여러 장의 연재**가 되는 구조입니다.

**② 자산 활용 트래킹 (검수용)**
```
- 어떤 자산을 (asset_name, asset_kind)
- 몇 번 LLM 호출했는지 (call_count)
- 각 파이프라인 단계에서 어디에 쓰였는지 (stage: interpret/discover/assemble/generate)
```

프롬프트에 "활용 자산: Yura(char), combat(mod)"라고 나오지만, 실제로는 그 자산을 0번 돌리고 주제만 가지고 이야기를 만든 경우도 있을 수 있습니다. 이걸 **투명하게 검수할 수 있게** 기록합니다.

**③ DB 스키마 (신규)**
```sql
-- chapter: experiment에서 뻗어나가는 각 장
CREATE TABLE chapters (
  chapter_id     VARCHAR2(64) PRIMARY KEY,
  exp_id         VARCHAR2(64) REFERENCES experiments(exp_id),
  chapter_num    NUMBER,              -- 1장, 2장...
  title          VARCHAR2(500),
  summary        CLOB,                -- 장 요약
  created_at     TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- scene: chapter 내의 개별 씬
CREATE TABLE scenes (
  scene_id       VARCHAR2(64) PRIMARY KEY,
  chapter_id     VARCHAR2(64) REFERENCES chapters(chapter_id),
  scene_num      NUMBER,              -- 1씬, 2씬...
  scene_type     VARCHAR2(50),        -- '전투', '대화', '탐색', '연출'
  title          VARCHAR2(500),
  content_full   CLOB,                -- 씬 전체 본문
  characters     CLOB,                -- 참여 캐릭터 (JSON)
  created_at     TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- asset_usage: 자산별 LLM 호출 횟수 트래킹
CREATE TABLE asset_usage (
  id             VARCHAR2(64) PRIMARY KEY,
  exp_id         VARCHAR2(64) REFERENCES experiments(exp_id),
  asset_name     VARCHAR2(500),
  asset_kind     VARCHAR2(50),        -- 'character', 'module', 'prompt'
  stage          VARCHAR2(50),        -- 'interpret', 'discover', 'assemble', 'generate'
  call_count     NUMBER DEFAULT 1,    -- 몇 번 LLM 호출했는지
  created_at     TIMESTAMP DEFAULT SYSTIMESTAMP
);
```
