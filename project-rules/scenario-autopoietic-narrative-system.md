# 자생 서사 시스템 — 초기 설계 draft

> **시나리오 팀**(`@heav_lnx_scenario_bot` → `~/projects/scenario`) 운영 대상.
> `@~/projects/scenario`(참고 자료 repo)의 RisuAI 4부품 조립법을 바탕으로, **"운영 가능한 세계 상태"** 를 만드는 시스템 설계.
> 이사님 지시(2026-07-01): *"자생 서사를 잘 뽑을 수 있게 준비해둔 게 repo. 지지고 볶아 만드는 **서사들은 구분해서 기록**해."*

## 0. 한 줄 정의

**자생 서사 시스템** = 캐릭터·세계가 상호작용 속에서 자극적으로 생성·운영되는, **상태를 가진** 서사 생태계.
산출은 "한 편의 이야기"가 아니라 **"다음에도 이어 굴릴 수 있는 세계 상태"** 다.

## 1. 핵심 철학 (세 기둥)

1. **상태 없는 엔진 계승** — LLM은 상태 없는 함수(`@~/projects/scenario/ecosystem/overview.md` §3). 모든 기억·세계관은 **매 턴 프롬프트에 다시 주입**. 그래서 "상태를 가진다" = "다음 턴에 주입할 세계 상태를 우리가 보존한다"는 뜻.
2. **자생(autopoietic)** — 세계·캐릭터가 미리 쓴 대본이 아니라 **상호작용에서 창발**. 운영자는 이야기를 쓰지 않고, *자라게 할 조건*을 설계한다.
3. **구분 기록(★ 이사님 지시)** — 지지고 볶아 만들어진 **각 서사는 서로 섞이지 않게 독립 canon**으로 기록.

## 2. 서사 인스턴스 분리 — "구분해서 기록"의 핵심 (★)

하나의 서사 = 하나의 **월드 인스턴스(world-instance)**. 인스턴스끼리 기본 차단.

- **월드 ID 체계:** `<세계관코드>-<런코드>` (예: `hunter-A01`, `kairos-B03`, `dating-app-X01`). 같은 세계관의 다른 런도 별개.
- **같은 캐릭터, 다른 canon:** "한지원"이 헌터물(`hunter-A01`)과 데이팅앱(`dating-app-X01`)에 모두 등장해도 → **각각 별도 character memory·상태**. 한쪽 사건이 다른 쪽에 묻지 않음.
- **크로스오버 금지(기본):** 인스턴스 간 캐릭터·설정 이동은 **명시적 수출 절차**(`§8`)로만.
- **정합 범위:** canon 정책은 *한 인스턴스 안에서만* 유효. 충돌 판정도 인스턴스 내부에서.

## 3. canonical 기록 5종 (인스턴스별 1세트)

| 기록 | 내용 | 대응(repo 4부품) |
|---|---|---|
| **world state** | 현재 세계 상황(시간·장소·공개 사실·긴장관) | 로어북(상시·발동) |
| **character memory** | 캐릭터별 목표·금기·관계·지금까지 겪은 일 | 캐릭터 카드 + 내장 로어북 |
| **canon policy** | 이 인스턴스의 불변 룰·금기선·정합 기준 | 시스템 프롬프트(제약부) |
| **event log** | 발생한 사건의 타임라인(append-only) | 채팅 로그의 정형 요약 |
| **retcon log** | 정정 이력(뭘 빼/바꿨고 왜) | (repo엔 대응 없음 — 운영 전용) |

## 4. 상태 라이프사이클 — 5단계

```
draft → observed → canon → deprecated → retconned
```

- **draft**: 첫 제안(설정/사건/캐릭터). 아직 세계에 반영 안 됨.
- **observed**: 한 번 이상 서사에 등장해 *관측*됨. 정합 검증 대기.
- **canon**: 정합 통과, 인스턴스의 확정 사실. 다음 턴 주입 우선순위 최상.
- **deprecated**: canon이었으나 폐기(사용 중단, 하지만 역사는 보존).
- **retconned**: 과거 사실을 **되감아 정정**. retcon log에 사유·대체본 필수.

> canon은 "진실"이 아니라 "현재 이 인스턴스가 동의하는 정합 상태" — 언제든 retcon 될 수 있고, 그 이력이 남는다.

## 5. 캐릭터 제약 + 재사용 우선

각 캐릭터는 3종 제약을 달고 운영된다(=캐릭터 카드의 정체성 보호):

- **목표(want):** 이 캐릭터가 지금 원하는 것. 행동 원동력.
- **금기(never):** 절대 안 하는 것/당하지 않는 것. 정체성 파괴 방어.
- **말투(voice):** 어조·어휘·버릇. 평균수렴 방지.

**재사용 우선 정책:** 새 캐릭터/설정을 만들기 전에, 기존 자산(`@~/projects/scenario/examples/`, `catalog/`)에서 재사용 가능한지 먼저 찾는다(persona §6 절차 준용). `templates/character-sheet.md` 등 빈 틀을 복사해 인스턴스 맥락에 맞게 치만 채운다.

## 6. 평가지표 (운영 품질)

| 지표 | 의미 | 건강 신호 |
|---|---|---|
| **충돌률** | canon 간 모순 발생 빈도 | 낮을수록 좋음 |
| **반복률** | 같은 전개/대사 반복 | 낮을수록 |
| **구분성** | 인스턴스 간 섞임(리백) 발생 | 0에 가까워야 |
| **retcon 빈도** | 정정 횟수 | 잦으면 canon 설계 불안 신호 |

> workflow.md §4 "한 번에 한 변수만" 원칙 준용 — 지표 악화 시 원인 매핑 후 자산 1개씩 수정.

## 7. repo ↔ 운영 데이터 — 양방향 피드백 루프 (이사님 07-01 확정, 정정)

> **정정:** 당초 "repo=불변 레퍼런스, 운영자 수정 자제"로 설계했으나 **철회**.
> repo는 살아있는 자산 — 이사님: *"그 과정 피드백으로 레포도 정리하거나 수정해 나가는거야."*

- **repo(`@~/projects/scenario`)** = 실전 자산(캐릭터 카드·로어북·프롬·모듈·예시) 창고. 운영 데이터(`worlds/`)도 같이 둔다(이사님 결정).
- **양방향 루프(★):** ① repo 실전 자산의 **기법**을 뽑아 자생 서사 시나리오에 **돌려 녹이고**, ② 그 적용에서 발견한 빈틈·부족 패턴·새 기법을 **피드백으로 repo에 정리·수정**.
- 게임/서사 개발 ↔ repo 정리가 서로 먹여주는 **self-improving 루프**(`@~/notes/principles/ai-dev-신념.md` §4 Closed loop · §7 문서=렌더링).
- repo 수정도 pull→commit→push 규칙 준수.

## 8. RPG 분리 + ADR 수출 (타 프로젝트 공유)

- **기본 분리:** RPG(`@~/projects/rpg_game`) 세계관은 RPG 팀 소유. 시나리오 팀 인스턴스와 자동 섞임 없음.
- **수출 절차(제한):** 어느 인스턴스의 자산(캐릭터·설정)을 RPG로 보내려면 → ADR로 자산·범위·라이선스 명시 후 수출. RPG 측이 수락해야 RPG canon 편입.
- **방향:** 시나리오 → RPG 단방향 수출 후보. 역방향(RPG→시나리오)은 별도 합의.
