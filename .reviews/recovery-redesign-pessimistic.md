# 복구번들 재설계안 — 비관론(pessimistic) 리뷰

> 대상: 계층형 + git 기반 복구번들 재설계안 (L1 매 턴 ~1.5K / L2 복구 1회 / L3 온디맨드)
> 관점: 리스크 · 실패 모드 · 놓친 점 · 역효과
> 작성: 2026-07-01 (매니저 리뷰 에이전트)
> 조사 기반 파일:
> - `~/.claude/hooks/cokacdir-recovery-gate.sh` (활성 게이트, 96줄)
> - `~/scripts/recovery-health-check.sh` (일일 점검, MAX_BUNDLE=9000 하드코딩)
> - `~/.claude/settings.json` (SessionStart/UserPromptSubmit 훅 바인딩)
> - `~/notes/decisions/2026-06-30-recovery-gate-bundle-shrink.md` (직전 ADR)
> - 번들 실측: MEMORY.md 1523B + current-work-state.md 2506B + work-queue.md 6189B = **10218B**

## 0. 전제 자체의 불안정 (가장 먼저 지적)

재설계안과 현장 코드 사이에 **한계값 불일치 3중**이 있다. 이것부터 정리하지 않으면
"다이어트"가 무엇을 위한 것인지 흔들린다.

| 출처 | 한계값 |
|---|---|
| 재설계안 현황 서술 | ">9000B 한계" (9000 고정 전제) |
| 활성 게이트 스크립트 `MAX_CONTEXT_BYTES` 기본 | **20000** (`COKACDIR_RECOVERY_MAX_BYTES` env 오버라이드 가능) |
| `recovery-health-check.sh` | `MAX_BUNDLE=9000` 하드코딩 |
| 직전 ADR(06-30) `.fail` 메시지 | "10503B > 9000B" |

즉 (a) 9000이었다가 게이트는 20000으로 올렸고, (b) health-check는 9000에 머물러 있고,
(c) 재설계안은 여전히 9000 기준으로 서술한다. **한계값 자체가 진동 중**이다.
이 상태에서 "계층 분리로 9000 이하를 만들자"는 목표는 모래 위의 집 — 한계를 20000으로
두면 현재 10218B는 **통과**한다 (게이트 관점). 그러면 재설계 전체의 동기가 사라진다.
**비관 결론 #0: 한계값을 먼저 확정(단일 소스)하지 않으면 재설계는 과잉해결일 수 있다.**

## 1. [위험 최고] L1만 매 턴 주입 → 매니저 판단 정보 누락

이것이 재설계안의 가장 치명적 약점이다.

**work-queue.md(6189B)가 실제 담는 것** — 단순 "큐"가 아니라 매니저의 턴 단위
판단 근거 전부:
- 🔥 **활성 작업** (autotrader 백테스트 진행/결과, RPG 컨셉 수렴, 시나리오 팀 신설)
- **대기 ADR** (라오어 단독/하이브리드/타방향/추가조사 — 아침 결정 대기)
- 🔥 **야간 자율 운영 승인경계선** (자동=사전합의실험·로컬계산·draft vs 승인필수=전략채택·commit·ADR·실거래)
- **오늘 밤 작업 2건** (exit_ratio 스윕, 엔진 ADR 근거표) — cron이 밤에 이것을 실행
- **다음 스텝** (v2 지시, 사칙 인증, 쿼터 대응)

**재설계안은 이것을 L2(/clear 후 1회)로 옮기자는 것.** 그런데:

1. **cron 야간 세션은 /clear 없이 SessionStart로 시작**한다. 즉 "L2 = /clear 후 1회"라는
   트리거는 cron 세션에서는 **발화하지 않는다**. cron 세션은 work-queue가 없으면
   "오늘 밤 작업 2건"을 모른 채 빈 프롬프트로 시작 → 야간 자율 운영이 돌지 않는다.
   **이것이 현 문제의 정확한 재현이다.** 재설계안은 자신이 풀려는 버그를 재생산한다.

2. **승인경계선 누락 = 실거래/commit 위험.** work-queue가 없는 매니저가 "야간이니
   자율로 진행하자"고 판단해 commit/push/전략채택에 들어가면 승인경계 위반이다.
   L1의 "핵심 3~5줄"에 승인경계를 요약할 수 있을까? 위 항목들을 3~5줄로 압축하면
   **정보 손실이 승인위반으로 직결**되는 항목들이 빠진다.

3. **"매니저 판단 저하"는 부드러운 표현이 아니다.** CLAUDE.md가 "매니저는 실행
   금지, 보고/조율만"을 규정하는데, work-queue 없이는 무엇을 조율할지조차 모른다.
   팀장 3명에게 뭘 지시할지 모르는 매니저 = 조직 마비.

**비관 결론 #1: L1-only 주입은 cron 야간 작업을 정확히 죽이고(원 버그 재현),
승인경계 누락으로 권한 침범 위험을 만든다. work-queue는 "상세"가 아니라 "핵심"이다.**

## 2. 온디맨드 Read — 실패·지연·비용의 3중고

재설계안: "L3 필요시 개별 ADR/문서를 온디맨드 Read. 상세는 git(notes repo)에서."

**실패 모드:**

- **게이트가 막힌 상태에서 Read 불가.** UserPromptSubmit 게이트는 `.ok`가 없으면
  `block_prompt`로 프롬프트를 suppress 한다(line 89-92). 이 상태에서 매니저가
  "일단 ADR을 Read하겠다"는 **도구 호출 자체가 프롬프트로서 차단**된다.
  즉 L3 온디맨드 Read가 작동하려면 이미 게이트가 풀려 있어야 하는데, 게이트가
  풀리려면 L1 주입이 성공해야 한다 — 순환. L1만으로는 work-queue 누락(위 #1)이므로,
  "L1 주입 → .ok → 이제 L2/L3 Read" 경로만 살아있는데, 그 경로는 **/clear 후에만**
  L2를 읽는다는 설계와 충돌한다 (cron은 /clear 아님).

- **지연 = 야간 윈도우 소진.** cron 23:00 트리거 → 매니저 세션 기동 → 게이트 통과 →
  work-queue Read(1회) → ADR Read(각각) → ... 매 Read마다 모델 왕복이 추가된다.
  야간 작업 2건을 끝내려면 L2+L3 Read가 5~8회. 한 번의 Read는 수천 토큰 +
  수 초. **막상 다이어트로 아낀 3KB(0.3%)를 Read 누적이 초과**할 수 있다
  (재설계안 자신이 "0.3%는 무의미"라고 인정했으므로, 이 비용 비교는 설득력 있음).

- **git 읽기 = 추가 실패 표면.** "상세는 git에서"는 notes repo가 clean/pull 됐다고
  가정한다. Windows 에이전트와 공유 중이므로 로컬이 뒤쳐지면 **과거 상태를 복구**로
  읽는다. 복구 번들의 가치는 "이 순간의 단일 진실"인데, git Read는 "마지막 pull
  시점의 진실"로 변질. CLAUDE.md "작업 전 git pull" 규칙이 매 턴 강제되지 않는 한
  (현재 강제 아님) 위험.

- **경로 외부 참조(@경로)의 신뢰성.** CLAUDE.md는 "외부 참조는 @경로 인라인으로
  충분한가?" 자문을 요구한다. L1이 "어디서 읽을지 파일 포인터"만 주면, 매니저가
  그 포인터를 **실제로 따라갈지**는 보장 없다. 모델은 종종 "맥락에 없으면 모른다"
  로 행동한다. 포인터는 강제가 아니다.

**비관 결론 #2: 온디맨드는 게이트 순환종속(Read하려면 .ok 필요, .ok는 L1만) +
야간 시간소모 + git 동기화 위험 + 포인터 무시 가능성의 합성 리스크.**

## 3. SessionStart/recovery-gate 스크립트 수정 복잡도 + 기존 세션 호환성

현재 스크립트는 **단일 경로 단일 주입**: `sessionstart` 모드 → 3파일 cat →
`emit_session_context $BUNDLE` → `.ok`. 약 96줄, 단순.

재설계안은 이것을 **계층 분기**로 바꿔야 한다:
- L1 주입(sessionstart) + work-queue/ADR은 주입 **안 함**(L1만)
- L2(/clear) 트리거 감지 → 별도 주입 또는 매니저에게 "Read하라" 지시
- cron 세션(source=startup, /clear 아님)은 L2를 **어떻게** 받을 것인가?

**복잡도 폭발 지점:**

- **source 매칭 분기.** settings.json의 SessionStart 훅은 matcher가
  `startup|clear|resume|compact` 4종이 **전부 동일 스크립트**로 바인딩.
  "L2 = /clear 후 1회"를 구현하려면 스크립트가 `$SOURCE`를 검사해 clear일 때만
  L2 번들을 추가 주입하거나, 별도 훅을 만들어야 한다. **그러면 cron(startup)과
  /clear(clear)가 다른 복구 깊이**를 갖게 된다 — 바로 #1의 cron 야간 작업 사망.

- **.ok 마커 의미 변화.** 현재 `.ok` = "3파일 주입 완료". L1-only가 되면
  `.ok` = "L1만 주입됨, work-queue는 안 읽음"이 되는데, UserPromptSubmit는
  여전히 `.ok` 존재만 본다(line 89). **L2를 안 읽었어도 게이트가 풀린다.**
  즉 "복구 메모리 로드 전 응답 금지"라는 게이트의 존재 이유가 무력화된다.
  매니저가 work-queue 모른 채 응답하는 것을 게이트가 **방조**한다.

- **기존 .ok 마커 잔류.** `recovery-gate/`에 9개 .ok가 있다. 재설계 전환 후
  이 마커들이 "구 의미(L1+L2+L3 주입됨)"로 남아 있으면, 새 스키마(L1만)에서
  UserPromptSubmit가 통과시켜 버린다. 마이그레이션 시 전체 삭제·재생성 필요.

- **health-check.sh 동기화.** 이 스크립트는 여전히 `MAX_BUNDLE=9000` + 3파일 합산.
  계층화하면 health-check도 "L1만 잰다"로 바꿔야 하는데, 그러면 번들 초과 조기경계가
  사라진다 (L1은 작으니까). **감시망이 사라진다.**

**비관 결론 #3: 단일 스크립트를 다계층으로 고치면 source 분기 + .ok 의미 재정의 +
마이그레이션 + health-check 재작성이 한 번에 터진다. 단순성 손실이 크다.**

## 4. "2계층 분리"가 단순성을 해치는가 — Yes, 그리고 더 위험한 결과

CLAUDE.md SELF-CHECK: "범위 최소? 파일 비대 금지." 재설계안은 이 원칙을
**인프라 레이어에서 위반**한다.

- **단일 진실(single source) 파괴.** 현재는 "복구 = 3파일"로 하나의 정의.
  계층화하면 "복구 = L1 + (조건부)L2 + (조건부)L3" — **복구의 정의 자체가
  상태 기계**가 된다. "이 세션은 복구됐는가?"에 대한 답이 yes/no가 아니라
  "어느 계층까지"가 된다. 디버깅 시 "왜 매니저가 ADR을 모르지?"를 추적하려면
  어느 계층이 주입됐는지 로그를 봐야 한다. 현재는 `.ok` 유무 하나.

- **사칙(원칙 체계) 인증 의무 약화.** 게이트 BUNDLE 헤더는 "아래 파일들로 사칙
  상속·인증 후에만 응답"을 강제. L1만 주입되면 사칙 인증의 **재료가 부족**하다
  (MEMORY.md 인덱스만). onboarding.md, principles/ 를 L3 온디맨드로 미루면,
  매니저가 사칙을 **인증했다고 착각**한 채 응답할 수 있다. CLAUDE.md가
  "읽기 전 결정/발판 금지"를 정한 취지가 침식.

- **팀장 3봇과의 비대칭.** 게이트는 매니저(cwd `.cokacdir/workspace/*`)에만
  적용, 팀장은 no-op(line 34-37). 그런데 매니저가 팀장에게 "이렇게 하라"고
  지시하려면 work-queue가 필요. L1-only 매니저가 팀장에게 **불완전 지시**를
  내리고, 팀장은 게이트 없이 그대로 실행 → 오류가 검출 안 됨(팀장 게이트 없음).
  계층화는 **매니저 약화 → 팀장 오류 전파** 경로를 만든다.

**비관 결론 #4: 계층 분리는 "복구됨"의 정의를 상태기계로 만들고, 사칙 인증·
팀장 통제의 단일 통로를 분산시켜 검증 가능성을 떨어뜨린다.**

## 5. 게이트 자체를 다이어트 말고 다르게 푸는 게 나은가 — 5가지 대안과 평가

재설계안의 근본 가정은 "번들을 줄여 게이트를 통과시킨다"인데, **게이트 통과를
번들 크기와 분리**하는 편이 더 견고할 수 있다.

### 대안 A — `MAX_CONTEXT_BYTES` 를 사실 기반 상향 (20000 이미 적용됨)
현재 활성 스크립트는 이미 20000. 현재 번들 10218B는 **이미 통과**.
- **문제:** health-check(9000)와 불일치, 그리고 재설계안의 9000 전제가 오래됨.
- **평가:** **가장 단순.** 한계값 단일화(전부 20000 또는 전부 같은 env)만으로
  현재 증상 해결 가능. 다이어트·계층화 불필요. 단, Claude Code가
  `additionalContext`를 일정 크기에서 preview로 대체하는지의 **실제 임계**를
  모르면 20000도 도박. 이 임계를 먼저 측정해야 한다 (현재 아무 문서도 측정값 제시 X).

### 대안 B — work-queue.md **자동 요약 더블**(원본 보존 + 요약본 주입)
work-queue.md 옆에 `work-queue.summary.md`(수백 B)를 두고, 게이트는 요약본만 주입.
원본은 온디맨드. 이것이 사실 재설계안 L1/L2의 **안전한 버전**.
- **평가:** 요약본 작성·유지 비용(이중 쓰기). 그러나 단일 스크립트 유지 + cron도
  요약본 받음(야간 작업 살림). **재설계안보다 우세.** 단 "요약본이 진짜 핵심을
  담는가" 검증 필요 (#1의 우려 그대로).

### 대안 C — 게이트를 "파일 존재"에서 "내용 해시 일관성"으로
번들 크기와 무관하게, MEMORY/state/queue가 마지막 known-good 해시와 같으면 .ok.
- **평가:** 크기 한계를 아예 제거. 그러나 Claude Code preview 대체 문제는
  별개(크기가 아니라 additionalContext 처리 한계). 근본 해결 아님.

### 대안 D — SessionStart 주입을 포기하고 UserPromptSubmit 첫 턴에 주입
SessionStart의 additionalContext preview 위험을 회피. 첫 UserPromptSubmit에서
번들을 additionalContext로 주입하고 그 턴은 통과.
- **평가:** 첫 프롬프트가 번들과 함께 가므로 "읽기 전 결정 금지"가 첫 턴에
  위반될 수 있다. 게이트 취지와 충돌. **탈락.**

### 대안 E — 번들을 다이어트하지 말고 **work-queue를 쪼개어 파일로**
work-queue의 "세션 노트/과거 상세"를 별도 `work-archive.md`로 빼고(이미 06-30 ADR이
부분 시도), work-queue는 "현재 활성 + 대기결정 + 다음스텝"만. 6189 → ~3000B 가능.
- **평가:** **게이트 스크립트 1줄 안 고침.** health-check만 한계 조정. 계층화 없이
  단일 경로 유지. cron 야간 작업 보존. **재설계안보다 단순·안전.**
  현재 work-queue 6189B 중 실제로 "세션 노트(요약)" 같은 아카이브성이 있어
  분리 가능. 이것이 다이어트의 **올바른 방향**.

**비관 결론 #5: 재설계안(계층화)보다 대안 A(한계 단일화) 또는 E(work-queue 아카이브
분리)가 같은 목적을 훨씬 적은 복잡도로 달성한다. 계층화는 over-engineering.**

---

## 종합 평결 (비관)

재설계안은 **세 가지 치명적 역효과**를 갖는다:

1. **원 버그 재현** — cron 야간 세션은 /clear가 아니므로 L2(/clear 후 1회) 트리거가
   발화하지 않아 work-queue가 주입되지 않고, 이것이 곧 현재 "야간 cron 안 돔" 증상.
   재설계안이 자신이 고치려는 버그를 재생산.

2. **게이트 무력화** — L1-only 주입 후 `.ok`가 남으면 UserPromptSubmit가 통과시키지만,
   매니저는 work-queue(승인경계·대기ADR·야간작업)를 모른다. "복구 전 응답 금지"라는
   게이트 존재 이유가 hollow 해짐.

3. **복잡도·단순성 교환 불균형** — 절약 0.3% 컨텍스트(재설계안 자신이 부정)를 위해
   단일 스크립트를 다계층 상태기계로 만들고, health-check·마이그레이션·사칙인증
   흐름까지 전부 재설계. 비용/편익이 역전.

**추천(비관 관점에서의 건의):**
- 즉시: `MAX_CONTEXT_BYTES` 와 health-check `MAX_BUNDLE` 을 **동일값으로 단일화**
  (둘 다 env `COKACDIR_RECOVERY_MAX_BYTES` 참조). 이것만으로 현재 10218B는 통과.
- 단기: work-queue.md에서 아카이브성("세션 노트 요약")을 `work-archive.md`로 분리
  (대안 E). 게이트 스크립트 수정 최소.
- **재설계안(계층화)는 보류** — 특히 cron 세션이 /clear가 아니라는 사실이
  "L2 = /clear 후" 설계를 정면으로 깨므로, 채택 시 야간 자율운영이 영구 사망.

## 부록 — 검증되지 않은 가정 (재설계안이 전제하나 입증 못 한 것)

- **"additionalContext가 N바이트에서 preview로 대체된다"** — Claude Code의 실제
  임계 미측정. 9000? 20000? 이 숫자의 근거가 어디에도 없다. ADR 06-30도
  "대체할 수 있음"(추측)으로 서술. **임계 측정이 선행되어야** 다이어트 목표치가
  의미를 갖는다.
- **"L1 1.5K 토큰이면 복구에 충분하다"** — 매니저의 턴 단위 의사결정에 필요한
  최소 정보량을 측정한 적 없음. work-queue만 6189B(약 1.5K 토큰)인데 이것 빼고
  "충분"이라는 주장은 근거 없음.
- **"git에서 읽으면 최신이다"** — Windows 에이전트 공유 repo의 동기화 상태
  미검증. CLAUDE.md "pull→push" 규칙이 매 턴 강제가 아님.
