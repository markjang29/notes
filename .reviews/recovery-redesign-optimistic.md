# 복구번들 재설계안 — 낙관론(optimistic) 리뷰

> 대상: cokacdir 복구번들 계층형(L1/L2/L3) + git 기반 재설계안
> 관점: 이점·기회·단순화·확장성 (optimistic)
> 검증자: 매니저 봇 리뷰 에이전트 (glm-5.2[1m])
> 날짜: 2026-07-01
> 메타: 번들 현황 = MEMORY 1523B + state 2506B + work-queue 6189B = 10218B (health-check MAX 9000B 초과, gate hook MAX 20000B 내). 차단의 실제 원인 = Claude Code 자체 additionalContext 한계(~9KB에서 preview 대체)와 health-check 9000B 기준의 불일치.

---

## 1. 리뷰 초점 5항목 — optimistic 평가

### (1) gate 해제로 cron 자동화 정상 복구 ★

**강한 긍정.** 번들을 L1(~1.5K 토큰 ≈ 4-5KB 텍스트)로 축소하면 Claude Code의 additionalContext 한계(~9KB preview 임계) 안으로 안정적으로 들어온다. 현재 가장 가치 있는 이점.

구체적 이득:
- **SessionStart `.ok` 마커 정상 생성** → UserPromptSubmit 차단 해제 → cron 세션(`cokacdir --cron --session`)이 즉시 의사결정 가능 상태로 진입.
- **health-check.sh MAX_BUNDLE=9000B 기준 통과** → 매일 점검 리포트가 "⚠️ 이상"에서 "✅ 세션 재시작 안전"으로 전환. 이사님 Telegram 보고 품질 향상.
- **야간 자율 운영 사이클(01:00 야간 / 07:00 아침 브리프) 복구** — 현재 work-queue.md §야간자율에 예약된 cron이 게이트 차단으로 무력화되어 있을 가능성. L1 다이어트 = 이 사이클의 잠금 해제.
- **reaper 스크립트와 시너지**: reaper는 유령 세션(529 원인)을 정리하고, L1은 정상 세션의 부팅을 보장 — 둘이 합쳐 "야간 자동화 파이프라인"의 양 끝을 닫는다.

**정합성 검증(0.3% of 1M 주장):** 1.5K 토큰은 1M 창의 0.15%. 설계안의 0.3%는 토큰-바이트 환산(토큰≈3-4바이트)을 감안하면 4-6KB 텍스트 = 1.5-2K 토큰 = 0.15-0.2%. 설계안 수치는 보수적(안전 측)이며, 실제로는 더 작을 수 있다. 다이어트 목적(gate 해제)에 충분히 부합.

### (2) git 기반 = 단일 진실원(source of truth) 일치 ★

**구조적 우위.** 현재 번들의 3 파일이 2개 저장소에 분산:
- `MEMORY.md`, `current-work-state.md` → akl0hdys 메모리 디렉토리(비-git, 세션 로컬 성격)
- `work-queue.md` → `~/notes`(git 추적, push 대상)

이 분산은 이미 정합성 리스크를 품고 있다. 재설계안의 "상세는 git(notes)에서 읽고 번들은 최소" 방향은 **진실원을 git 하나로 수렴**시킨다.

이점:
- **Windows 다중 에이전트 협업 정합성**: CLAUDE.md 규칙("작업 전 git pull, 작업 후 add/commit/push")에 따라 Windows 머신의 에이전트가 같은 notes repo를 공유. 번들 상세가 git에 있으면 Windows 측에서도 동일한 진실원 접근 가능 — 세션 로컬 메모리(akl0hdys)와의 불일치 제거.
- **버전 이력 = 복구 백업**: work-queue.md 변경 이력이 git log에 남아, "이전 상태로 롤백"이 자연스럽게 지원. 현재는 세션 로컬 메모리 분실 시 복구 불가.
- **clear-recovery-map과 정렬**: 기존 맵("남음: ~/notes는 git push, 세션과 무관")이 이미 git을 영속층으로 지정. 재설계안은 이 기존 원칙의 논리적 귀결.
- **context-explosion-causes 원칙 강화**: "서브에이전트 결과는 파일로 덤프, 메인엔 결론만" 원칙이 복구 번들에도 동일 적용 — 일관된 아키텍처.

**기회**: akl0hdys 메모리 디렉토리 전체를 notes repo의 `memory/` 서브디렉토리로 흡수하면(이사님 승인 시) 단일 git 저장소 = 단일 진실원 완성. 이미 `~/notes/memory/` 디렉토리가 존재함(빈 듯) — 자연스러운 이주 지점.

### (3) L1/L2/L3 계층 — 재사용 가능한 패턴인가 ★

**강한 재사용성.** 이 패턴은 cokacdir 매니저 봇에 국한되지 않는다. 범용 복구 아키텍처로 타 프로젝트(RPG/autotrader/scenario 팀장 봇) 및 외부 프로젝트에 적용 가능.

왜 재사용 가능한가:
- **의존성 최소**: L1은 "인덱스 + 핵심 상태 + 파일 포인터"라는 추상 패턴. 특정 파일명에 종속되지 않음. 각 프로젝트는 자신의 MEMORY 인덱스 + 현재 상태 + 포인터만 제공하면 됨.
- **지연 로딩(lazy loading) 원칙**: L2/L3는 "필요할 때만 Read" = 정보 공학의 표준 계층 설계. 캐시(L1) → 디스크(L2) → 아카이브(L3) 구조와 동형.
- **스케일링 대응**: 프로젝트가 커지며 산출물이 늘어나도 L1은 일정 크기로 고정(인덱스만). 현재 flat 구조는 산출물 증가 → 번들 비대 → 게이트 재차단 루프. 계층화는 이 루프를 구조적으로 차단.
- **팀장 봟 적용 경로**: 현재 게이트는 매니저(cwd `~/.cokacdir/workspace/*`)에만 적용, 팀장은 no-op. 하지만 팀장 봇도 `/clear` 후 복구가 필요. L1/L2/L3 템플릿을 팀장별로 인스턴스화(rpg_game/autotrader/scenario 각각)하면 팀장 복구 게이트도 동일 패턴으로 확장 가능.

**기회 — 패턴 공식화**: 이 계층을 `~/notes/principles/recovery-layering.md`로 문서화하면, 향후 신규 프로젝트 온보딩 시 "복구 번들 = L1 인덱스 + L2 큐 + L3 디테일" 표준으로 채택 가능. principles/ 디렉토리(이미 context-budget.md v2 등 운영규칙 보관)의 자연스러운 확장.

### (4) 매 턴 주입 축소 — 비용/지연 이점 ★

**직접적 비용 절감.** 현재 매 SessionStart(startup/clear/resume/compact 4개 matcher)마다 10218B가 additionalContext로 주입. 4개 이벤트 × 9-10KB = 세션 수명주기당 36-40KB 재주입.

L1(~4-5KB)으로 축소 시:
- **주입량 ~50% 절감**: 10218B → ~4500B. compact/resume 빈도가 높을수록 누적 절감.
- **첫 응답 지연 단축**: SessionStart 주입은 첫 턴 응답 시작 전 처리. 주입 분량이 작을수록 첫 토큰까지의 지연 감소. cron 야간 자동화(지연에 민감)에서 특히 유리.
- **context-explosion-causes 원칙 5(측정)와 시너지**: token-report Stop 훅이 매 턴 누적을 출력. 번들 축소는 베이스라인 컨텍스트를 낮춰, 실제 작업에 쓸 수 있는 가용 예산 증가. 1M 창에서 6KB 절감은 미미해 보이나, compact 후 재주입 시마다 복리 효과.
- **의사결정 품질 간접 향상**: 더 작은 베이스라인 = 더 많은 유효 컨텍스트 = 더 긴 작업 세션 가능 = 더 깊은 검증 루프 수용(기존 context-budget 원칙 3 "검증 루프 세션 분산"의 여유 확보).

**정량 근거**: 10218B ≈ 2.5-3.4K 토큰(토큰≈3-4바이트). L1 1.5K 토큰으로 축소 시 약 1-1.9K 토큰 절감/주입. compact 3회 세션이면 ~3-5.7K 토큰 누적 절감. 작지만 비용 0인 구조적 이득.

### (5) work-queue "매 턴" → "복구 시" 이동 — 의사결정 품질 영향 ★

**가장 흥미로운 개선.** work-queue.md(6189B, 번들의 60%)를 매 턴 주입에서 제거하는 것은 단순 다이어트가 아니라 **정보 신선도(freshness) 관리**다.

왜 의사결정에 유리한가:
- **work-queue는 변경 빈도가 낮음**: 활성 작업·대기 결정은 세션마다 바뀌지 않는다(마지막 갱신 06-30 14:10, 현재 07-01). 매 턴 재주입은 비용 대비 가치가 낮은 정보. 복구 시 1회 Read면 충분 — 그 시점이 가장 신선해야 할 순간.
- **L1은 핵심 인덱스만 → 인지 부하 감소**: 매 턴 6189B의 큐 전문이 주입되면, 모델은 모든 활성 작업(autotrader 백테스트 + RPG 컨셉 + 시나리오 팀 + 아이디에이션)을 동시에 고려해야 한다. L1이 "현재 포커스 1개 + 포인터"만 주면, 모델은 현재 작업에 집중하고 다른 작업은 온디맨드(L3)로 가져온다 — 의사결정의 범위 좁힘 = 품질 향상.
- **work-queue 노후화 리스크 감소**: 큐가 매 턴 주입되면 "항상 최신"이라는 착각을 준다. 하지만 실제로는 수동 갱신(마지막 06-30). "복구 시 Read"로 바뀌면, Read하는 행위 자체가 "이 시점의 스냅샷"임을 명확히 함 — 신선도 검증(mtime/git log)을 자연스럽게 유발.
- **L2 복구 시점의 의사결정 집중**: 복구 = "어디까지 했고 다음이 뭔가"를 결정하는 순간. 이때만 work-queue를 읽으면, 큐가 의사결정에 직접 기여하는 순간에만 로드 — 정보의 목적-적합성(purpose-fitness) 극대화.

**잠재 우려(optimistic 내에서의 미세 조정)**: work-queue를 복구 시에만 읽으면, 긴 세션 중 큐가 갱신되었을 때(팀장이 push) 매니저가 인지 못할 수 있다. 그러나 이는 L1의 "파일 포인터"가 git 최신 커밋 해시나 mtime을 포함하면 해결 — 포인터가 "큐 변경 감지" 트리거 역할. optimistic 관점에서 이는 설계안의 자연스러운 확장점이지 결함이 아니다.

## 2. 부수적 이점 (5항목 외)

### 단순화 효과
- **health-check.sh와 gate hook 기준 불일치 해소**: 현재 health-check=9000B, gate hook=20000B로 기준이 갈라져 있어 판정이 엇갈림(10218B는 health-check 위반, gate hook 통과). L1이 9000B 이하로 설계되면 두 기준 모두 자동 통과 — 기준 통합 불필요, 자연 해소.
- **MEMORY.md 인덱스 역할 명확화**: 현재 MEMORY.md는 1523B로 이미 인덱스 성격. 재설계안은 이를 L1의 핵심으로 공식화 — 기존 구조의 의도를 명시적으로 승격.

### 확장성 — 미래 시나리오
- **시나리오 팀(06-30 신설) 온보딩**: 새 팀장 봇이 같은 L1/L2/L3 패턴으로 복구 게이트 획득 가능. 패턴이 있으면 온보딩 비용 감소.
- **프로젝트 수 증가 대응**: 현재 3팀(rpg/autotrader/scenario). 4-5팀으로 확장 시 flat 번들은 감당 불가. 계층화는 선형적이지 않지만 L1 고정 크기로 O(1) 확장.
- **AI Native 원칙 체계(principles/)와 통합**: 복구 계층화를 원칙으로 승격하면, 향후 에이전트 온보딩(scratch onboarding) 시 "복구 아키텍처 이해"가 표준 단계로 편입.

### git 기반의 운영 이점
- **변경 추적 가능**: 번들 다이어트 자체가 git 커밋으로 기록. "언제 무엇을 축소했나" 감사 가능.
- **롤백 안전망**: L1 축소 후 문제 발생 시 git revert로 이전 번들 복원. 세션 로컬이면 불가능.

## 3. 구현 기회 — 즉시 취할 수 있는 윈

1. **work-queue.md 다이어트(6189B → 분리)**: 현재 work-queue는 활성 작업(상세) + 대기 결정 + 야간 스케줄 + 세션 노트가 한 파일. L2에서는 "활성 작업 헤드라인 + 대기 결정"만, 상세 백테스트 결과/내러티브는 L3 개별 문서(이미 `.reviews/`에 존재)로 분리. 포인터만 L2에 남김.
2. **current-work-state.md(2506B) → L1 인라인**: 이 파일은 이미 L1 성격(조직 + 진행 + 대기 결정 요약). 그대로 L1 채택, 약간 축소.
3. **MEMORY.md(1523B) → L1 인덱스**: 이미 인덱스. 그대로 L1.
4. **포인터 강화**: L1에 "최근 git 커밋 해시 + work-queue mtime" 추가 → 신선도 감지 트리거. ~100B 추가만으로 L3 온디맨드 로드 품질 향상.

예상 L1 크기: MEMORY 1523B + state(축소) ~1500B + 포인터 ~200B = ~3200B ≈ 0.8-1K 토큰. 9000B 기준에 여유. 20000B 기준에도 여유. Claude Code additionalContext 한계에도 안전.

## 4. 총평 (optimistic)

재설계안은 **현재의 명확한 문제(gate 차단 → cron 마비)를 구조적으로 해결**하면서, 동시에 **단일 진실원(git), 재사용 가능한 계층 패턴, 비용 절감, 의사결정 품질 향상**의 4가지 부수적 이득을 가져온다. 

가장 강한 점: 이 재설계는 **새로운 복잡성을 추가하지 않는다** — 기존 자산(MEMORY 인덱스, git notes, `.reviews/` 분리 원칙)의 역할을 명확히 재배치할 뿐. context-budget.md와 clear-recovery-map이라는 기존 원칙의 자연스러운 실천이다. 

위험이 아닌 기회 관점에서, 이 재설계는 cokacdir 복구 시스템을 "단일 프로젝트용 임시방편"에서 "다중 에이전트 조직의 표준 복구 아키텍처"로 승격시키는 전환점이다.
