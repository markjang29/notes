### 원칙 2. 서브에이전트 프롬프트 — 다이어트

- §2는 "맥락·이전 비교·문서 전문을 프롬프트에 넣지 않는다"와 "파일 경로만 전달 → 에이전트가 직접 읽음"을 제시한다. 이 방식은 메인 재적재는 줄이지만, 서브에이전트가 같은 큰 파일을 매번 full read하는 비용과 서브에이전트 내부 컨텍스트 폭발은 막지 못한다. §13은 "서브에이전트 큰 반환값의 메인 재적재 + 루프 누적"을 주범으로 지목하지만, 프롬프트 다이어트만으로는 루프마다 파일 전문을 새로 읽는 누적을 다루지 않는다.
- "파일 경로만"은 리뷰 품질에 필요한 범위 지정과 충돌한다. 이번 참고 맥락처럼 "markjang29.md (§1-4, §10)"가 필요한 경우 단순 경로만 전달하면 에이전트가 불필요한 섹션까지 읽거나, 반대로 중요한 섹션을 놓칠 수 있다. 경로와 함께 섹션 범위 또는 line range를 전달하는 규칙이 필요하다.
- markjang29.md §4는 의도 왜곡 감지 시 "목표 재진술 → 원안 ↔ 변경점 분리 → 국소 수정 중단"을 요구한다. 그런데 §2의 "맥락·이전 비교" 금지는 원안과 변경점 비교에 필요한 최소 맥락까지 금지하는 문구로 읽힐 수 있다. "전문 금지"와 "판정에 필요한 최소 비교 맥락 허용"을 분리하지 않으면 정지 프로토콜과 충돌한다.

### 원칙 3. 검증 루프 — 세션 분산

- §3은 "비판-수용 2회 루프를 한 세션에 몰지 않는다"와 "산출물 단위로 세션 분리, 또는 compact 사이에 1회씩"이라고 한다. 이 원칙은 §13의 "루프 누적"을 다루지만, "산출물 단위"와 "compact 사이에 1회씩"의 선택 기준이 없다. 결과적으로 운영자는 같은 루프를 세션 분리할지 compact만 할지 매번 판단해야 하고, 판단 기준 부재가 루프 누적을 다시 허용한다.
- "비판-수용 2회 루프"만 언급되어 3회 이상 반복되는 논쟁, 추가 검증, 회귀 리뷰 같은 실제 누적 상황이 규칙 밖에 놓인다. 2회 초과 시 강제 파일화/새 세션/요약 인계 같은 상한이 없다.
- markjang29.md §1은 완성 조건으로 "현재 목표 1–2줄 · 중단 지점 · 다음 스텝(작업 큐)"를 요구한다. §3의 세션 분산은 이 인계 정보 없이는 맥락 손실을 만든다. §3에는 세션 분리 직전 반드시 남길 handoff 형식이 없고, §43의 "compact 손실(맥락 누락)보다 낫다"는 주장만 있다.
- markjang29.md §4의 중량 정지 프로토콜은 "작업 큐·도구 인벤토리 갱신"과 "진행 재개 전 확인"을 요구한다. §3은 세션 전환 비용을 인정하지만, 전환 시 작업 큐나 검증 대기 상태를 갱신하는 요구가 없어 정지 프로토콜과 연결되지 않는다.

### 원칙 4. Read 절제

- §4는 "같은/큰 파일 full 재독 금지", "`offset/limit` 부분 읽기, 한 번 읽은 것은 재활용"이라고 한다. 하지만 "큰 파일"의 기준, "같은 파일"의 범위, "재활용" 가능한 기간이 없다. 이 상태에서는 200줄 문서, 2,000줄 문서, 직전 세션에서 읽은 문서가 모두 운영자 판단에 맡겨진다.
- "한 번 읽은 것은 재활용"은 markjang29.md §4의 의도 왜곡 감지와 긴장 관계가 있다. 원안이 바뀌었거나 사용자가 범위를 바꿨는데 과거 읽기 결과를 재활용하면 "원안 ↔ 변경점 분리"가 틀릴 수 있다. 재활용 전 파일 mtime/hash 확인, 섹션 번호 확인, 사용자 최신 요청 우선 같은 무효화 조건이 필요하다.
- §4는 읽기량만 줄이고, 읽은 내용을 메인 답변에 얼마나 재삽입할지 제한하지 않는다. 컨텍스트 폭발은 full read 자체뿐 아니라 읽은 전문을 분석문에 길게 인용하거나 반복 요약할 때도 발생한다. 인용 길이, 섹션별 발췌 상한, "근거는 섹션 번호 우선" 같은 출력 제한이 빠져 있다.
- `offset/limit`는 특정 도구 명령에 종속된 표현이다. 실제 작업에서는 `sed -n`, `rg`, `nl`, MCP 리소스, 웹 문서 등 다양한 읽기 경로가 있으므로 "부분 읽기"의 추상 규칙과 도구별 예외가 분리되어야 한다.

### 원칙 5. 측정 — token-report

- §5는 "Stop 훅(`~/.claude/skills/token-report.sh`)이 매 응답 후 누적 토큰을 출력"한다고 한다. 문서의 대상은 "Codex/검증 에이전트"와 1M 컨텍스트 창인데, 측정 도구 경로는 `~/.claude/skills/`에 고정되어 있다. Codex 세션에서도 해당 훅이 실제로 동작한다는 근거가 문서 안에 없다.
- "임계(예: 60%) 도달 시 즉시 compact / 산출물 분리"는 사후 대응이다. §13에서 문제로 지목한 "서브에이전트 큰 반환값의 메인 재적재"는 한 번의 반환으로 임계치를 크게 넘길 수 있으므로, 응답 후 측정만으로는 예방이 아니라 사후 수습에 그친다.
- 60%라는 예시는 1M 창 기준 약 600k 토큰까지 누적을 허용한다. 사용자가 지적한 "1M 토큰 창 지연"을 막기에는 너무 늦을 수 있다는 위험이 있다. 문서에는 60%의 산출 근거, 경고/차단 2단계, 작업 유형별 더 낮은 임계가 없다.
- "즉시 compact / 산출물 분리"는 markjang29.md §1의 "이어받기 좋은 상태"와 충돌할 수 있다. compact 직전 현재 목표, 중단 지점, 다음 스텝을 남기라는 조건이 §5에 없어서, 토큰 절약이 아이디어·맥락 보존보다 앞설 수 있다.

## 누락된 원칙

- 반환 채널 분리 원칙이 없다. §1이 파일 저장을 요구해도 "원문 그대로는 사용자 표시용"이 메인 응답 전문 반환으로 해석되면 핵심 누출이 재발한다. 표준출력/채팅에는 경로와 짧은 요약만, 전문은 파일로만 제공한다는 규칙이 필요하다.
- 인계 요약 원칙이 없다. markjang29.md §1의 "현재 목표 1–2줄 · 중단 지점 · 다음 스텝"을 세션 분산·compact·리뷰 완료 시 의무화해야 한다. 지금 §3과 §5는 세션을 끊는 방법만 있고 이어받기 조건이 없다.
- 무효화/재읽기 기준이 없다. §4의 "한 번 읽은 것은 재활용"에는 파일 변경, 사용자 최신 요청, 섹션 변경, 리뷰 라운드 변경 시 재확인 조건이 붙어야 한다.
- 서브에이전트 산출물 크기 예산이 없다. §1은 메인으로 올리지 말라고만 하고, 리뷰 파일 자체의 길이·구조·요약 섹션·근거 섹션 분리 기준은 없다. 산출물이 과도하게 길면 나중에 다시 읽을 때 같은 문제가 반복된다.
- 루프 상한과 탈출 기준이 없다. §3은 2회 루프를 언급하지만, 실패 반복 시 "수정 필요" 판정, 사용자 확인, 작업 큐 등록, 세션 종료 중 무엇을 택할지 정하지 않는다.
- `.reviews/` 인덱스 또는 최신본 참조 규칙이 없다. 같은 대상 리뷰가 쌓이면 어느 파일이 최신/승격/폐기 대상인지 찾기 위해 디렉토리 전체를 읽는 누출이 생길 수 있다.

## 과잉 원칙

- §2의 "파일 경로만 전달"은 과잉이다. 경로만으로는 필요한 섹션 범위, 판정 기준, 금지된 출력 방식이 전달되지 않는다. "파일 전문 금지"는 필요하지만, "경로 + 섹션 범위 + 산출물 계약 + 반환 상한"까지는 허용해야 한다.
- §3의 "산출물 단위로 세션 분리"는 작은 산출물에는 비용이 클 수 있다. markjang29.md §1도 "작은 작업은 필수 3개만 채워도 완성. 전부 채우기를 강제하지 않는다"고 하므로, 짧은 리뷰나 단일 파일 소규모 수정까지 세션 분리를 강제하면 인계 비용이 절약 효과를 넘을 수 있다.
- §4의 "같은/큰 파일 full 재독 금지"는 변경 가능성이 있는 원문에는 과잉으로 작동할 수 있다. 재활용은 필요하지만, 변경 여부 확인 없이 full 재독 금지만 적용하면 오래된 맥락으로 판단하는 비용이 커진다.

## `.reviews/` 운영 가능성 평가

- 저장 위치는 §1과 §36-37에서 일관되게 `~/notes/.reviews/`와 "Codex 리뷰 등 서브에이전트 결과물 보관"으로 제시된다. 하지만 운영 단위가 "리뷰", "서브에이전트 결과물", "승격/수용 완료" 중 무엇인지 불명확하다.
- 파일명 `<대상>-<YYYYMMDD>.md`는 같은 날 같은 대상에 대해 여러 작성자, 여러 라운드, 수정판이 있을 때 충돌한다. 이번 요청의 지정 파일명 `codex-context-budget-20260626.md`는 작성자와 대상을 포함하므로 기본 규칙보다 낫지만, 라운드/상태는 여전히 빠져 있다.
- "정기 폐기(승격/수용 완료 후)"는 폐기 기준으로 부족하다. markjang29.md §10은 "아이디어 원문/링크 살아있으면 통과; 사라졌으면 버그 수정 전 복원 우선"이라고 하므로, 폐기 전에 어떤 내용이 어디로 승격됐는지 링크가 필요하다. 현재 규칙대로 폐기하면 리뷰 원문이 유일한 아이디어 흔적일 때 §2의 "아이디어 손실" 위험에 걸린다.
- 보존 기간, 상태값, 소유자, 승격 대상 링크, 폐기 전 체크리스트가 없다. 이 부재는 `.reviews/`가 장기적으로 전체 재독을 유발하는 저장소가 되거나, 반대로 필요한 비판 근거를 너무 빨리 삭제하는 양쪽 위험을 만든다.

- 충돌: context-budget.md §2의 "맥락·이전 비교·문서 전문을 프롬프트에 넣지 않는다"는 markjang29.md §4의 "원안 ↔ 변경점 분리"와 충돌할 수 있다. 원안/변경점 분리에는 최소 비교 맥락이 필요하다.
- 충돌: context-budget.md §5의 "임계(예: 60%) 도달 시 즉시 compact"는 markjang29.md §1의 완성판정과 §4의 정지 프로토콜을 건너뛸 수 있다. compact 전에 목표/중단점/다음 스텝과 원안/변경점 분리가 남아야 한다.
- 충돌: `.reviews/`의 "정기 폐기(승격/수용 완료 후)"는 markjang29.md §10의 "아이디어 원문/링크 살아있으면 통과; 사라졌으면 버그 수정 전 복원 우선"과 조건부로 충돌한다. 승격 위치 링크 없이 폐기하면 원문/링크 생존을 검증할 수 없다.
- 불일치: markjang29.md §10은 "개요만 보고 다음 행동을 정할 수 있으면 통과"라고 하지만, context-budget.md의 5줄 요약 규칙에는 "다음 행동" 포함 의무가 없다. 요약이 결론만 담으면 리뷰판정 기준을 통과하지 못한다.

## 수치 기준(5줄 등) 타당성

- "결론 요약(5줄 이내)"는 메인 반환값을 억제하는 수치로는 의미가 있으나, 모든 산출물에 고정 적용하기에는 근거가 없다. 원칙 1-5 각각의 상태를 한 줄씩만 써도 5줄이 소진되어 파일 경로, 다음 스텝, 위험도, 미해결 질문이 빠진다.
- markjang29.md §1의 필수 완성 요소는 "현재 목표 1–2줄 · 중단 지점 · 다음 스텝"이다. 5줄 제한은 이 필수 요소를 담을 수는 있지만, 리뷰 결과의 핵심 비판 3-5개까지 동시에 담으려면 부족하다. 따라서 "5줄"보다는 "파일 경로 1줄 + 핵심 비판 최대 5개 + 다음 행동 1줄"처럼 필드 기준이 더 운영 가능하다.
- §5의 "임계(예: 60%)"는 예시로만 표시되어 실제 운영 기준으로 약하다. 1M 창에서 60%는 이미 매우 큰 누적이고, 지연을 막으려면 경고 임계와 차단 임계를 분리해야 한다. 예: 30% 경고, 45% 신규 전문 금지, 60% 강제 compact 같은 단계화가 필요하다. 이 수치는 제안이며, 현재 문서에는 산정 근거가 없다.
- "비판-수용 2회 루프"는 실제 검증 루프를 설명하는 숫자지만, 2회 이후의 처리 기준이 없다. 2회를 넘으면 새 세션, 작업 큐 등록, 사용자 확인 중 하나로 전환한다는 종료 조건이 필요하다.

## 우선순위별 개선 권고

1. §1에 반환 채널 규칙을 추가하라. "전문은 파일에만 저장, 메인/표준출력은 파일 경로 + 핵심 요약 + 다음 행동만"이라고 명시해야 "원문 그대로" 문구가 재적재 경로가 되지 않는다.
2. §3과 §5에 markjang29.md §1의 인계 필드를 의무화하라. 세션 분리나 compact 직전 "현재 목표 1–2줄, 중단 지점, 다음 스텝, 참조 파일"을 남기지 않으면 컨텍스트 절약이 맥락 손실로 바뀐다.
3. §2를 "파일 경로만"에서 "파일 경로 + 필요한 섹션/라인 범위 + 산출물 계약 + 반환 상한"으로 바꾸라. 전문 프롬프트 금지는 유지하되, 검증에 필요한 최소 비교 맥락은 허용해야 한다.
4. `.reviews/` 규칙에 충돌 없는 파일명과 retention 메타데이터를 넣어라. 권장 형식은 `<agent>-<target>-<YYYYMMDD>-r<N>.md` 또는 지정 파일명 + front matter `status`, `supersedes`, `promoted_to`, `delete_after`이다.
5. §4에 재활용 무효화 기준과 인용/출력 상한을 추가하라. 파일 변경, 범위 변경, 사용자 최신 요청 변경 시 재확인하고, 근거는 섹션 번호 우선·짧은 인용 보조로 제한해야 한다.
